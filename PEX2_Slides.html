Data Structure: Process

Defined in processQueue.h. Each process carries all the information your CPU thread needs.

typedef struct { int PID; // unique identifier (0, 1, 2, ...) int arrivalTime; // timestep it joined the readyQ int priority; // 0-5; LOWER number = HIGHER priority int burstTotal; // total CPU time needed (used by SJF) int burstRemaining; // time left -- decrement this each timestep int initialWait; // timesteps waiting before first run (auto-tracked) int totalWait; // total timesteps spent in readyQ (auto-tracked) bool requeued; // set to true when YOU put it back in readyQ } Process;

Fields you actively use:
• burstRemaining — decrement once per timestep
• burstTotal — original burst length; equals burstRemaining on first scheduling
• priority — read by NPP and PP
• requeued — set true when preempting/requeueing

Don't forget requeued!
Whenever you insert a process back into readyQ (Round Robin quantum expiry, SRTF preemption, PP preemption), set p->requeued = true before calling qInsert.
This field affects initialWait tracking.

Data Structures: SharedVars & CpuParams

Defined in CPUs.h. Everything shared between the main thread and all CPU threads lives in SharedVars.

typedef struct { pthread_mutex_t readyQLock; pthread_mutex_t finishedQLock; sem_t** cpuSems; sem_t* mainSem; Queue readyQ; Queue finishedQ; int quantum; } SharedVars; typedef struct { int threadNumber; SharedVars* svars; } CpuParams;

Key fields

  • readyQLock / finishedQLock — always lock before touching a queue, unlock immediately after
  • cpuSems[i] — CPU i waits on this at the start of each timestep; main posts it
  • mainSem — CPU posts this when done; main waits for one post per CPU
  • readyQ / finishedQ — the shared doubly-linked queues
  • quantum — only used by RRcpu
  • threadNumber — which CPU you are; used to index into cpuSems

Extracting your parameters at the top of every CPU function:

int threadNum = ((CpuParams*) param)->threadNumber; SharedVars* svars = ((CpuParams*) param)->svars;

Synchronization: The Clock Loop

The main thread IS the simulation clock. Each timestep follows a strict handshake with every CPU thread.

Step 1
Step 1 — Generate: Main thread optionally creates a new process and inserts it into readyQ.

Your responsibility in every CPU function: (1) call sem_wait(svars->cpuSems[threadNum]) at the top of the loop, (2) do one timestep of work, (3) call sem_post(svars->mainSem) at the bottom. Never skip the post — that deadlocks the simulation.

The Queue API — Functions You Will Call

All declared in processQueue.h. You will use these directly in CPUs.c.

Core operations

// Append to tail (always inserts at end) void qInsert(Queue* q, Process* proc); // Remove the node at 0-based position 'which' // Returns NULL if queue is empty or out of bounds Process* qRemove(Queue* q, int which);

Position selectors (return index for qRemove)

// Position of the process with the shortest burstRemaining int qShortest(Queue* q); // Position of the highest-priority process (lowest number) int qPriority(Queue* q);

Value helpers (return a scalar, for comparisons)

// Smallest burstRemaining value in the queue int qShortestBR(Queue* q); // Best (lowest) priority number in the queue int qGetPriority(Queue* q);

The mutex pattern (every queue access)

pthread_mutex_lock(&(svars->readyQLock)); p = qRemove(&(svars->readyQ), 0); if (p == NULL) { printf("No process to schedule\n"); } else { printf("Scheduling PID %d\n", p->PID); } pthread_mutex_unlock(&(svars->readyQLock));

Print INSIDE the lock. The printf for scheduling/no-process must happen while you hold the mutex. Hold the lock for the minimum time needed, then release it before executing the process.

Both selectors return 0 for empty queues. Always check if qRemove returned NULL before using p.

FIFO — The Given Implementation (Read This Carefully)

This is the complete FIFOcpu function. Every other algorithm follows this exact structure.

void* FIFOcpu(void* param) { int threadNum = ((CpuParams*) param)->threadNumber; SharedVars* svars = ((CpuParams*) param)->svars; // p is the process currently running on this CPU. // p == NULL means the CPU is idle and must pick a new process from readyQ. Process* p = NULL; // This thread runs forever — one loop iteration = one simulation timestep. while (1) { // —— Sync point 1: wait for main to start this timestep —————— // main() posts cpuSems[threadNum] once per tick. We block here // until that post arrives, keeping this CPU in lockstep with the clock. sem_wait(svars->cpuSems[threadNum]); // —— Selection (only when idle) ——————————————————— // FIFO is non-preemptive: once a process is running (p != NULL) we // never replace it mid-burst. We only enter this block when the CPU // has nothing to run. if (p == NULL) { // Lock readyQ before inspecting or modifying it — another CPU // thread (or main inserting a new arrival) could touch it right now. pthread_mutex_lock(&(svars->readyQLock)); // Index 0 = head of the list = the process that has been waiting // the longest (qInsert always appends to the tail, so the head is // always the oldest arrival — that is the FIFO selection rule). p = qRemove(&(svars->readyQ), 0); if (p == NULL) { // readyQ was empty — CPU stays idle this tick. printf("No process to schedule\n"); } else { printf("Scheduling PID %d\n", p->PID); } pthread_mutex_unlock(&(svars->readyQLock)); } // —— Execution: one unit of work ————————————————— // If we have a process (carried over from a prior tick or just // selected above), burn one unit of its remaining CPU burst. if (p != NULL) { p->burstRemaining--; if (p->burstRemaining == 0) { // Process is done — move it to finishedQ so main can // compute and print wait-time statistics at simulation end. pthread_mutex_lock(&(svars->finishedQLock)); qInsert(&(svars->finishedQ), p); pthread_mutex_unlock(&(svars->finishedQLock)); // CPU is now idle; it will select a new process next tick. p = NULL; } } // —— Sync point 2: signal main that this CPU is done ————— // main() waits on mainSem once per CPU per tick. Posting here // tells main this CPU has finished its work for the current timestep. sem_post(svars->mainSem); } }

The Generic CPU Thread Pattern

Every CPU function builds on the same synchronization skeleton given in FIFOcpu. The differences between algorithms are conceptual — figure out the code structure by reading what you have.

Selection criterion
Which process is picked from readyQ when the CPU is idle. This changes in every algorithm.

Quantum rotation (RR only)
A running process may be returned to readyQ after a fixed number of timesteps, even if it is not finished.

Preemption (SRTF & PP only)
A running process may be displaced mid-burst when a “better” process appears in readyQ.

Start from FIFOcpu. Read every line until you understand it fully. Then ask: what is the minimum change needed to make this function behave like SJF? Like NPP? Like RR?

All Six Algorithms at a Glance

Algorithm Preemptive? Selection criterion When does preemption occur?
FIFO No Earliest arrival (head of queue) —
SJF No Shortest total burst —
NPP No Highest priority (lowest number) —
RR Quantum Earliest arrival (head of queue) After every quantum timesteps
SRTF Yes Shortest remaining burst When a shorter job arrives in the queue
PP Yes Highest priority (lowest number) When a strictly higher-priority job arrives

For non-preemptive (FIFO, SJF, NPP):
Only select when p == NULL. Once a process is running, it stays until it finishes.

For preemptive (SRTF, PP):
The check-and-select logic runs every timestep, not just when the CPU is idle. Think carefully about your locking strategy.

Before the simulation starts, the program prints a table of all processes that will arrive during the run. This goes to stderr so it does not affect graded output.

How to use it

  • The table is identical for all algorithms — same seed, same arrivals
  • Before running the sim, trace your algorithm by hand using this table
  • Verify: which process should be running at each timestep?
  • Check the debug output (debug=1) against your hand trace

Example analysis for FIFO

  • t=0: PID 0 arrives (burst=2). Scheduled immediately.
  • t=1: PID 0 runs (burst 1 left).
  • t=2: PID 0 finishes. Queue empty.
  • t=5: PID 1 arrives (burst=2). Scheduled.
  • t=8: PID 2 arrives (burst=8). PID 1 still running...

Try tracing SJF, NPP, or RR (q=3) using the same table before coding them up.

Key implementation detail: At t=11, when the quantum expires, PID 3 was already in readyQ (main inserted it before signaling the CPU). So after requeuing PID 2 at the tail, readyQ = [PID 3, PID 2]. The head (PID 3) is selected next. This ordering matters for correct output.

Recommended Plan of Attack

  1. Read FIFOcpu until you understand every line. Do not write any code yet. Use the process table (slide 11) to trace through it step by step by hand.
  2. Implement the non-preemptive algorithms first (SJF, NPP). They differ from FIFO only in which process is selected from the ready queue. Run grade.sh after each one.
  3. Implement RR. Selection is the same as FIFO, but you need to handle quantum expiry. Think carefully about what state you need to track across timesteps.
  4. Implement the preemptive algorithms (SRTF, PP). These require an additional check every timestep — not just when the CPU is idle. Think carefully about your locking strategy.
  5. Run grade.sh after each algorithm. A passing test confirms correctness before you move on.

Start with FIFO passing the grader (it should, out of the box), then commit. Commit after every algorithm that passes.

The debug flag (1 as first argument) prints the full queue state at the start of every timestep. Use it together with the process table to trace exactly which process should be running when.

Common Pitfalls

Deadlock

Forgetting sem_post(mainSem) at the end of the loop will freeze the entire simulation. The main thread waits forever for a post that never comes.

Missing requeued = true

Whenever you put a process back into readyQ (RR, SRTF, PP), set p->requeued = true before calling qInsert. Missing this corrupts initialWait statistics.

Race condition in SRTF/PP

If you check whether to preempt, release the lock, then re-lock to call qRemove, another CPU can steal the process in between. Keep a single lock across both operations.

Printing outside the lock

The printf for "Scheduling PID X" / "No process to schedule" must happen while holding readyQLock, so output order is consistent with queue state.

Segfault from NULL dereference

qRemove returns NULL for an empty queue. Always check before using p. Add -fsanitize=address to your compile flags to get better diagnostics.

Useful debug flags

-Wall -Werror — required for submission
-fsanitize=address — finds memory bugs
-g — enables debugger symbols for gdb
